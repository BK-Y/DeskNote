# DeskNote 文档说明

本文件只说明 **当前 `docs\` 目录的真实用途**，避免把旧计划稿、学习笔记和当前执行计划混在一起。

## 当前文档分工

### 1. 当前有效入口

- `README.md`：项目当前状态、构建方式、源码结构
- `docs\design\architecture.md`：当前架构边界与长期演进约束
- `.ai\plan.md`：当前执行中的总计划入口
- `.ai\phases\`：当前各阶段详细计划
- `.ai\standards\phase-planning.md`：阶段读取 / 制定标准

### 2. `docs\` 目录的定位

- `docs\design\`：设计文档与历史设计草稿
- `docs\notes\`：学习笔记、历史排查记录、技术备忘
- `docs\plan\`：**旧版计划文档归档**，保留作回溯参考，不再作为当前实施依据

## 需要特别注意的状态

### `docs\plan\`

这里的 `PLAN_*` 与 `progress.md` 主要记录的是早期分阶段想法。  
**当前真正执行的阶段计划已经迁移到 `.ai\` 目录。**

### `docs\design\PL-*.md`

这些文件是较早的专题规划 / 设计草稿，保留作历史参考。  
如果其中内容与当前代码、当前 `.ai\` 计划冲突，**以当前代码和 `.ai\` 计划为准**。

## 更新规则

### 必须同步更新的情况

当出现下面任一变更时，提交里应同步更新对应文档：

1. 源码目录结构变化 → 更新 `README.md`
2. 模块边界 / 分层职责变化 → 更新 `docs\design\architecture.md`
3. 当前阶段目标变化 → 更新 `.ai\plan.md` 与对应 `.ai\phases\phase-*.md`
4. 某份旧文档已不再适合作当前依据 → 在文档顶部明确标注“历史归档 / 非当前执行依据”

### 不应继续做的事情

- 不要再把当前执行计划优先写进 `docs\plan\`
- 不要引用不存在的 `docs\adr\` 目录作为当前规范
- 不要让 `README.md` 长期停留在旧源码结构或旧功能状态

## 维护原则

- 文档统一使用 Markdown
- 面向当前开发的说明，优先写在 `README.md`、`docs\design\architecture.md`、`.ai\`
- 历史材料可以保留，但必须让读者一眼看出它是不是“当前有效”

## 备注

如果未来确实需要正式的 ADR 体系，可以再新增 `docs\adr\`。  
在那之前，`docs\design\` 与 `.ai\standards\` 就是当前更实际的设计承载位置。
