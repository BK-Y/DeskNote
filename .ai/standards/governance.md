# Project Governance

## 一、通用硬约束（违反即不合格）

1. **计划先行** — 任何代码变更必须先有对应的书面计划，计划经审查确认合规后方可实施。不允许"先写代码再补计划"或"边写边计划"。计划的粒度可以是纲领或详细计划，但不能没有

2. **严格解耦** — 不把本应分开的职责临时糊在一起。"先这样写后面再拆"是不允许的默认策略

3. **严格分层** — 所有模块分层以 `.ai/arch.md`（主架构）和各子架构文档中定义的依赖规则为准。制定计划前，必须评估当前架构是否能覆盖计划目标所需的模块和分层关系；如果不能，必须先由 AI 产出架构修订方案、经用户确认后更新架构文档，再继续制定计划。不得在分层不明确的情况下"先写了再说"

4. **架构先行** — 每次新增模块或变更分层关系后，必须同步更新对应的架构文档。架构文档与源码不一致视为未完成。架构由 AI 给出修订方案，用户确认后实施

5. **禁止无控制跨层** — 各层只能通过对外接口与其他层通信。不得绕过接口直接访问其他层的内部结构。接口清单在各层对应的架构文档中定义

6. **已完成阶段冻结** — `_done` 文件只能被读取引用，不能被改写。后续变化必须通过新增 Phase 承接

## 二、工作流程

### 2.1 新建计划标准流程

从需求到一份合格的计划文件，按以下流程通过。

```text
需求/目标
  ↓
① 评估当前架构文档是否覆盖目标所需的模块和分层
  ├─ 能覆盖 → 进入步骤③
  └─ 不能覆盖 → ② AI 出架构修订方案 → 用户确认 → 更新架构文档 → 进入步骤③
  ↓
③ 制定计划（按 4.2 的 8 关流程 + check list 逐项通过）
  ↓
④ 方案设计 → AI 列出候选方案 + 选型依据
  ↓
   ╔═══ 人机决策 ═══╗
   ║ 人确认方案方向  ║
   ╚════════════════╝
  ↓
⑤ 继续完成计划余下关卡（④拆解执行 → ⑤设定边界 → ⑥落地方案 → ⑦验收标准 → ⑧推进步骤）
  ↓
⑥ 审查（对照架构文档检查分层、接口、调用限制）
  ↓
⑦ 用户确认完整计划
  ↓
   计划文件就绪
```

### 2.2 审核计划标准流程

当需要审核一份已有计划时，按以下流程逆向走一遍。审核的目的是**修复缺陷**，不是机械打勾。

```text
收到审核指令
  ↓
① 读待审核的计划文件
  ↓
② 从⑧推进步骤开始逆向检查：
    ┌───────────────────────────────────────────────┐
    │ 审核六问 + 呈现检查（自底向上）：            │
    │                                               │
    │ 第六问：推进步骤可执行吗？                    │
    │   操作内容具体到文件/函数？结果可验证？       │
    │          ↓                                    │
    │ 第五问：落地方案能支撑推进步骤吗？            │
    │   技术路线覆盖所有操作？接口完整？            │
    │          ↓                                    │
    │ 第四问：方案设计是否合理？                    │
    │   方案符合架构分层规则？排除路径有说明？      │
    │   选型依据充分？如缺失则补全到计划文件        │
    │   如存在多方案，必须点名推荐方案并说明推荐    │
    │   原因——基于架构合规性、模块独立性和执行      │
    │   成本综合判断                                │
    │          ↓                                    │
    │   分层校验：从模块独立视角分析                 │
    │   如果后续阶段要改这个模块的实现，需要动哪些  │
    │   文件？如果动了已完成的文件，说明分层不当，  │
    │   应调整为后阶段只改本层文件                  │
    │          ↓                                    │
    │ 第三问：边界够清楚吗？                        │
    │   范围防膨胀？约束到位？                      │
    │          ↓                                    │
    │ 第二问：拆解和产出够具体吗？                  │
    │   依赖链填了？产出可观察？                    │
    │          ↓                                    │
    │ 第一问：问题定义对吗？                        │
    │   写的是"堵着什么"而不是"实现什么"？         │
    │          ↓                                    │
    │ 呈现检查（文档规范）：                        │
    │   ├─ 章节结构完整？没有漏掉关卡？             │
    │   ├─ 术语一致？枚举名/函数名前后统一？        │
    │   ├─ 代码无低级语法错误？                     │
    │   ├─ 推进步骤的验证方式与操作结果对应？       │
    │   └─ 引用的文件/函数名存在且正确？            │
    └───────────────────────────────────────────────┘
  ↓
③ 定位出问题的关卡
  ↓
④ 分析问题：解释为什么这是问题（架构冲突/未来阻塞/无法执行等）
  ↓
⑤ 给出修复方案：针对每个问题写出具体的修复方式
  ↓
⑥ 修复该关卡的内容
  ↓
⑦ 向后同步：该关卡之后的所有关卡如有依赖，一并调整
  ↓
⑧ 输出修复后的计划文件
  ↓
   如方案方向变更 → 回到人机决策点
   如无需变更方案方向 → 计划就绪
```

**审核修复原则：**
- 除非方案方向本身要变（换方案），否则不需要重新经过人机决策点
- 审核发现的问题必须附带：**问题是什么、为什么是问题、怎么修**。禁止只报问题不给方案
- 审核发现小缺陷（如漏填依赖、产出不具体）直接修复，不打断流程
- 审核发现方案方向有问题时，AI 提出修订后的候选方案，**必须点名推荐方案并说明推荐原因**，回到人机决策点由人确认
- **模块独立校验贯穿所有审核：** 任何一层未来的变更是否只需要改本层的文件？如果需要改已完成的文件，说明当前分层不当，必须调整

### 2.3 读取规则（从 plans.md 到实施入口）

1. 读 `plans.md`，找到当前焦点阶段
2. 读焦点对应的纲领文件
3. 纲领文件中列出子阶段列表，找到当前焦点的子阶段文件
4. 逐层递归，直到读到无子阶段列表的文件（即详细计划）
5. 读取该文件的⑧推进步骤，逐条执行

### 2.4 阶段完成收尾流程

当一个详细计划完成后：

```
a) 确认完成标准全部满足
b) 代码变更已编译通过（0 error 0 warning）
c) 计划文件重命名：{原文件名} → {原文件名}_done.md
d) 更新上级纲领文件的子阶段状态表（标记为"已完成"）
e) 更新 plans.md 状态（如果该阶段在 plans.md 中有记录）
f) git add + git commit + git push
```

纲领文件的所有子阶段完成后，纲领文件自身也重命名为 `_done.md`。

纯验证阶段（不修改代码）：跳过 b、f，只需重命名 + 更新状态表。

### 2.5 Repair 计划

当详细计划实施过程中发现需要修复的问题时，**另起一个 Repair 计划**，不修改原计划文件。

**适用场景：**
- 计划目标已实现，但边界情况处理不当
- 外部环境变化导致原实现不兼容
- 原实现的隐式假设被打破

**不适用场景：**
- 原计划本身的目标未完成（这是原计划没做完，不是 repair）
- 新功能需求（这是新 Phase，不是 repair）

**命名规则：** `{原计划文件名}_repair-{序号}.md`
- 例：`plan-10-shell-3a_1_repair-1.md`
- 已完成时重命名为 `_repair-1_done.md`

**Repair 计划适用的规则：**

| 规则 | 是否适用 | 说明 |
|------|---------|------|
| 一、硬约束（6 条） | ✅ 全部适用 | 解耦、分层、架构、跨层、冻结、计划先行一个不能少 |
| 新建计划流程（2.1） | ✅ 适用 | 评估架构→方案设计→人机决策→完成计划→审查→确认 |
| 阶段收尾流程（2.4） | ✅ 适用 | 完成后重命名 `_done` |
| 编号规则 | ⚠️ 不适用 | repair 用 `_repair-{N}` 固定后缀 |
| 计划要求（4.2） | ✅ 全部适用 | 按 8 关流程 + check list 逐项通过 |

## 三、计划体系结构

### 3.1 层次树

```text
plans.md（总计划）
  └── plan-{编号}.md（纲领文件）
        └── plan-{编号}-{字母}.md（子纲领文件）
              └── plan-{编号}-{字母}_{数字}.md（详细计划，最后一层）
```

### 3.2 编号规则

数字-字母-数字-字母…交替，不能连续两个数字或两个字母。
例：`1 → 1a → 1a_1 → 1a_1a`
数字段表示主要阶段，字母段表示子阶段。

### 3.3 plans.md 只记录主要阶段

只记录第一段数字，不展开子阶段。子阶段进度由各级纲领文件管理。

## 四、计划要求

> 本节写给**写计划的 AI**。目的是约束 AI 按固定流程和规范生成一份合格的计划文件。
> 计划文件的读者包括人和 AI，但本节的目标读者只是写计划的 AI——它在生成计划时必须完成以下所有关卡。

### 4.1 架构文档

架构文档的位置不固定，允许用户自定义。

**默认约定：**
- 主架构：`.ai/architecture.md` — 7 层定义、全局依赖规则、各模块接口规范
- 子架构：`.ai/sub_arch/arch-{phase}.md` — 该阶段涉及的模块归属、接口、调用限制

用户可在 `.ai/system.md` 中自定义架构文档的路径。AI 写计划时以 `.ai/system.md` 中的配置为准；如未配置，使用以上默认约定。

每个模块的条目应包含：

| 字段 | 说明 |
|------|------|
| **位置** | 文件路径 |
| **归属层** | 属于哪一层 |
| **职责** | 该模块负责什么 |
| **不负责** | 该模块不负责什么 |
| **对外接口** | 函数签名、宏定义、回调消息号。其他层只能通过这些接口与本模块通信 |
| **调用方式** | 调用顺序、生命周期、线程安全说明、返回值含义 |
| **调用方限制** | 谁可以调用、谁禁止调用 |
| **使用示例** | 标准调用代码片段（可直接复制使用） |
| **越界示例** | 禁止的操作（反面教材） |

### 4.2 计划要求

一份计划文件不预先区分为"纲领"还是"详细计划"。文件类型由内容自决：

- **包含子阶段列表 → 纲领**，负责拆解与分发，自身不直接实施
- **不包含子阶段列表 → 详细计划**，自身包含全部实施细节

子阶段列表的存在因为**有具体问题要解决**。问题驱动拆解，拆解产生子阶段，子阶段产生依赖关系。不先定义问题则没有拆解的必要。

#### 思考流程与约束关卡

写计划时按以下流程逐关通过，每关的约束不能跳过。

```
① 定义问题 ──→ 不满意 ←──┐
   ↓                      │
② 分析现状 ──→ 不满意 ←──┤
   ↓                      │
③ 方案设计 ──→ 不满意 ←──┤
   ↓                      │
   ╔═══ 人机决策 ═══╗     │
   ║ AI 提供方案     ║     │
   ║ 列出选型依据    ║     │
   ║ 人确认方向      ║     │
   ╚════════════════╝     │
   ↓                      │
④ 拆解执行 ──→ 不满意 ←──┤
   ↓                      │
⑤ 设定边界 ──→ 不满意 ←──┤
   ↓                      │
⑥ 落地方案 ──→ 不满意 ←──┤
   ↓                      │
⑦ 验收标准 ──→ 不满意 ←──┤
   ↓                      │
⑧ 推进步骤 ──→ 不满意 ←──┘
                (回溯到出问题的关卡修复)
```

**回溯规则：** 任何一个关卡完成后，如果发现前序关卡的信息不充分或决策有误，**允许且应当回溯到出问题的关卡重新处理**，而不是在当前关卡硬凑。禁止"先跳过，后面再补"。

##### ① 定义问题

本阶段要解决的关键问题是什么。

**约束：** 必须明确写出本阶段要解决的关键问题。禁止用"实现某某功能"代替问题。问题 = 当前有什么堵着 / 缺着 / 不对，要打通。功能是解法，不是问题。

| 关卡 | 纲领 | 详细计划 |
|------|------|---------|
| 核心问题 | ✅ 必填 | ✅ 必填 |

##### ② 分析现状

当前代码和架构处于什么状态，前置依赖是否到位。

**约束：**
- 承接关系必须写明前置阶段是谁、依赖什么、消费什么产出。禁止凭空假设前置条件已满足
- 阶段摘要一句话总括本阶段范围。详细计划必须包含 `before→after`

| 关卡 | 纲领 | 详细计划 |
|------|------|---------|
| 承接关系 | ✅ 必填 | 可省略（前置信息已内化） |
| 阶段摘要 | ✅ | ✅（详细计划含 before→after） |

##### ③ 方案设计

有几个可行的解法，各有什么依据。

**制定计划时：**
- AI 列出候选方案（至少 2 个，除非只有 1 条路可走）
- 每个方案必须写明：
  - 怎么做的概要思路
  - **是否符合当前架构的分层规则**（引用 `architecture.md` 或 `arch-*.md` 的具体条目）
  - 是否需要改动架构；如需要，说明改什么（新增模块、变更分层归属、调整调用限制等）
- **禁止给出违反现有分层/解耦约束的方案**
- 写明每个方案的选型依据和优缺点
- **从模块独立视角校验：** 如果后续阶段要改这个模块的实现，需要动哪些文件？如果动了已完成的文件，说明分层不当，必须重新设计
- AI **不做最终选择**——列出方案和依据后，由人决策选择方向
- **所有候选方案全部写入计划文件的③方案设计节**，选中的方案在行尾标记 `[selected]`
- 如需改动架构，须在人确认方向后、进入⑥落地方案前，先更新架构文档（规则见 4.3）

**审核计划时：**
- 审核已有计划文件中的方案是否合理：
  - 方案是否符合架构分层规则？是否考虑了其他可行路径？是否选型依据充分？
  - 如方案缺失 → 补全方案到计划文件的③方案设计节
  - 如方案不合理 → 在对话中提出修订方案，由人确认后更新

**仅列出单一路径时：** 即使只有一条可行路径，也要**在对话中**说明考虑了哪些被排除的方案及理由。证明确实考虑过其他可能，而非直接跳过。

| 关卡 | 纲领 | 详细计划 |
|------|------|---------|
| 方案选型（含依据，不含最终决策） | ✅（整体阶段方案选型） | ✅ 必填 |

##### ④ 拆解执行

怎么拆成可执行的步骤，做完后系统具备什么能力，能看到什么结果。

**约束：**
- 有子阶段列表则视为纲领。表格必须包含序号、状态、主题、主要改动文件、依赖关系。禁止把"模糊的大步骤"当子阶段
- 整体目标以 list 表达本阶段完结后系统具备的能力。禁止写抽象方向词
- 产出 = 做完后能看到/验证的具体结果。好的产出示例：新文件创建、函数新增、窗口出现按钮。**坏的产出示例：** 性能提升、体验改善、基础能力加强

| 关卡 | 纲领 | 详细计划 |
|------|------|---------|
| 子阶段列表 | ✅（有则视为纲领） | ❌（不存在） |
| 整体目标 | ✅（整体目标） | ✅（具体交付目标） |
| 阶段产出 | ✅（各子阶段的产出概要） | ✅（本阶段逐项可观察结果） |

##### ⑤ 设定边界

在什么范围内做，各层做什么，改哪些文件。

**约束：**
- 范围必须同时写"包括什么"和"不包括什么"。约束条件直接写在范围中（如：只能同步操作、不能新增依赖库等），不另开章节
- 各层分别做什么。禁止把不同层的逻辑堆在同一层
- 文件落点必须同时写"预计修改的文件"和"原则上不应修改的文件"。禁止让 AI 自己猜改哪个文件

| 关卡 | 纲领 | 详细计划 |
|------|------|---------|
| 本阶段范围 | ✅（概要，含范围+不做+约束） | ✅（详细，含范围+不做+约束） |
| 分层归属 | ✅（概要，子阶段涉及层） | ✅（详细，各层具体职责） |
| 文件落点 | ✅（概要清单） | ✅（具体到文件） |

##### ⑥ 落地方案

选中的方案如何落地到具体代码。

**约束：**
- 技术路线必须落到具体代码路径。必须包含主链路（最核心调用链用 `→` 图画出）。应覆盖：新增入口、状态承接、调度/适配、旧路径调整
- 接口说明仅在实现接口的阶段需要。必须写明：接口签名、调用方式、调用顺序、生命周期、返回值含义、使用示例

| 关卡 | 纲领 | 详细计划 |
|------|------|---------|
| 技术路线 | ❌（由子阶段展开） | ✅（含主链路） |
| 接口说明 | ❌（由子阶段展开） | ✅（条件：实现接口时） |

##### ⑦ 验收标准

怎么算做完。

**约束：** list 可验证、可验收的标准。必须包含一项"分层核实"（交叉验证分层归属与文件落点一致）。禁止写"稳定可用""基本完成"等模糊词。

| 关卡 | 纲领 | 详细计划 |
|------|------|---------|
| 完成标准 | ✅（纲领完成条件） | ✅（可验收标准，含分层核实） |

##### ⑧ 推进步骤

按什么顺序做，每一步动什么文件、怎么动、完成后验证什么。

**约束：**
- 推进步骤按选中的方案分段展开，**以选定方案名称为段落标题**，在该段落内列出执行序列
- 每段（每个方案）必须按顺序列出每一步要做什么。每步至少包含：
  - **操作内容**：改哪个文件的哪个函数 / 新增哪个文件
  - **操作结果**：这一步完成后系统处于什么状态（可验证）
  - **验证方式**：怎么确认这一步做对了（编译通过 / 函数可调用 / 窗口出现某元素）
- 推进步骤是 AI 执行的直接指令序列。AI 应严格按顺序逐条执行，完成一条再进入下一条
- 纲领态不填此项（子阶段的推进步骤在各自详细计划中展开）

示例：

```
### 方案 B：按职责拆为 5 个子阶段 [selected]

| 步骤 | 操作内容 | 操作结果 | 验证方式 |
|------|---------|---------|---------|
| 1 | `src/foo.c` 新增 `Bar_Init()` | Bar_Init 可被调用 | 编译通过 |
| 2 | ... | ... | ... |

### 方案 A：整体实现（备选）

（本方案未选中，推进步骤为空，留待后续切换时填充）
```

**切换方案时：** 只需要在新的方案段落中补充推进步骤，旧的推进步骤保留不动，不互相覆盖。

| 关卡 | 纲领 | 详细计划 |
|------|------|---------|
| 推进步骤 | ❌（由子阶段展开） | ✅ 必填 |

#### 接口描述的层级粒度

调用接口的描述在不同层级的计划中粒度不同：

| 层级 | 接口描述粒度 | 示例 |
|------|-------------|------|
| 纲领 | 只列涉及哪些模块、模块间的大致调用关系 | "Shell-3 涉及 tray 模块，app 调 Tray_Init/Tray_Destroy，window 接收 WM_TRAYICON" |
| 子纲领 | 列出接口签名、调用方向 | "Tray_Init(HWND) 由 app 在初始化后调用" |
| 详细计划 | 完整接口说明 + 调用方式 + 使用示例 | NOTIFYICONDATAW 初始化代码、调用位置、错误处理 |

**原则：接口的详细使用说明只出现在实现该接口的详细计划中。** 上级纲领只引用不展开。
详细计划中的接口说明应包含：调用顺序、生命周期、线程安全说明、返回值含义、标准调用代码片段。

### 4.3 计划与架构文档的关系

#### 架构文档定位

| 文件 | 内容 | 默认路径 | 谁维护 |
|------|------|---------|--------|
| 主架构 | 7 层定义、全局依赖规则、各模块接口规范 | `.ai/architecture.md` | AI 出修订方案，用户确认后更新 |
| 子架构 | 该阶段涉及的模块归属、接口、调用限制 | `.ai/sub_arch/arch-{phase}.md` | AI 在方案设计阶段产出 |

路径以 `.ai/system.md` 中自定义配置为准；未配置时使用以上默认路径。

#### 计划文件中涉及架构改动的处理规则

在③方案设计阶段，每个候选方案必须检查是否符合现有架构。根据检查结果分三种情况：

**情况 A：方案完全符合当前架构**

→ 直接进入④拆解执行，不碰架构文档。

**情况 B：方案需要新增子架构（如新增模块或接口，但不违反主架构）**

→ 在③方案设计中产出子架构内容。按以下规则决定放哪：

| 情况 | 放哪 |
|------|------|
| 子架构涉及新增模块或接口，后续阶段可能引用 | `.ai/sub_arch/arch-{phase}.md` |
| 子架构只是临时接口说明，阶段完成后不再有用 | ⑥落地方案中直接写明，不单独开文件 |

**情况 C：方案需要改动主架构（如变更分层归属、调整调用限制、新增跨层依赖）**

→ **必须先更新 `architecture.md`，再继续后续关卡。** 更新流程：
1. AI 出架构修订方案
2. 用户确认
3. 更新 `.ai/architecture.md`（或自定义路径的主架构文件）
4. 回到③方案设计，确认方案已与更新后的架构一致
5. 继续④拆解执行

> 禁止在未更新架构文档的情况下进入⑥落地方案。

## 五、使用规则

1. 新建计划前先读本文件，按 4.2 的 8 关流程逐关通过 + check list 逐项打勾
2. 审查时逐项对照架构文档检查分层和接口限制
3. 本文件与具体计划冲突时，先修计划，不绕开本文件
4. AI 给出任何变更方案后，需用户确认方可实施
