### 总目标要求

当前总目标至少包含下面 7 项要求：

1. **单窗口**
   - 当前不进入多窗口模型

2. **单便签**
   - 当前以单文档、单文件主链推进

3. **可输入**
   - 用户可以持续编辑正文内容

4. **可保存**
   - 编辑结果可持久化到磁盘

5. **可恢复**
   - 重启后可恢复上次内容

6. **严格解耦**
   - 在满足上面能力的同时，必须保持严格分层和边界稳定

7. **用户数据安全**
   - 不能把“正常退出时才保存成功”当作可接受最终状态
   - 异常退出、电源中断、保存中断等场景下，必须尽可能减少数据丢失，并避免正式文件被半写破坏

### 当前总目标不要求

当前总目标明确不要求：

- 多窗口
- 多文件切换
- Markdown 富文本渲染
- 富交互组件

### 产品形态补充（后续主线）

虽然当前基础目标仍是“单窗口、单便签、能输入、能保存、重启能恢复”，
但接下来的产品方向要开始收敛为**真正像便签**的窗口形态：

1. 窗口不再依赖系统标题栏与系统边框
2. 标题区改成自绘，并预留更多壳层命令按钮
3. 必须支持自定义拖拽、缩放与 hit test
4. Windows 上要支持托盘后台运行，不长期占用任务栏位置
5. “常驻桌面”拆成两种模式：
   - 浮动常驻最前
   - 贴边占位，最大化窗口不侵入

## 当前方案收束

当前只围绕 5 条主链推进：

1. 启动链：程序能启动
2. 显示链：窗口里能画出编辑区
3. 输入链：能输入、删除、移动光标
4. 存储链：能保存、能恢复
5. 壳层链：窗口形态、托盘后台、桌面常驻模式

### 当前对 render 的定义

`render/` 当前被定义为**基础绘制后端**：

- 负责：帧生命周期、矩形、文字、基础文本测量/命中
- 不负责：编辑规则、界面语义、输入事件、存储逻辑

一句话约定：

> `ui` 决定画什么，`render` 负责怎么画。

## 当前边界约定

- `app/`：组装层，负责初始化、生命周期、事件分发
- `platform/`：平台适配层，负责 Win32 窗口、消息循环、输入事件、系统 API
- `render/`：绘图后端，负责 Direct2D / DirectWrite 封装与基础绘图能力
- `ui/`：界面组件层，负责标题栏、编辑区、布局、显示状态
- `editor/`：编辑核心，负责插入删除、光标、选区、键盘命令
- `core/`：纯数据与语义，负责 `Document` 及 Markdown 相关数据结构
- `storage/`：持久化，负责 note 保存/加载、state、恢复

核心边界：

1. `core/` 不依赖 Win32 和渲染层
2. `render/` 不知道标题栏、光标、选区、标签等业务概念
3. `storage/` 不直接操作 UI，不持有窗口句柄
4. `platform/` 只翻译系统事件，不直接修改文档
5. `app/` 只做组装与调度，不堆积具体业务逻辑

## 后续保存能力路线

### 保存安全主线的统一内存模型

后续 Plan 6-9 统一围绕下面 3 个对象推进：

1. **工作文本（working text）**
   - 当前 `editor` 正在编辑的内存内容

2. **正式落盘基线（persisted baseline）**
   - 最后一次成功写入正式文件后的可信内容状态

3. **恢复层（recovery snapshot）**
   - 尚未进入正式主文件、但用于异常退出后兜底恢复的辅助内容层

统一约定：

- `dirty` 不再被视为孤立布尔值
- 它本质上表示：
  - **working text 已偏离 persisted baseline**
- Plan 7 负责决定“何时把 working text 推进到 persisted baseline”
- Plan 8 负责决定“推进 persisted baseline 时如何可靠写盘”
- Plan 9 负责决定“working text 尚未进入 persisted baseline 时，如何提供恢复层”

后续保存能力的推进顺序固定为：

```text
Plan 6 建立内存落盘模型
-> Plan 7 基于模型接入自动保存
-> Plan 8 基于模型接入原子保存
-> Plan 9 基于模型接入崩溃恢复
```

更后续再进入多文件切换保存策略。

### 当前保存链的风险结论

当前保存链虽然已经满足“关闭后能恢复”，但还不满足“用户数据安全优先”。

当前主要风险有 3 类：

1. **异常退出丢失内存修改**
   - 现在只在正常关闭时保存
   - 进程崩溃、系统重启、断电时，最近编辑内容会直接丢失

2. **正式文件可能被半写破坏**
   - `note.md` 和 `state.ini` 目前都还是直接 `CREATE_ALWAYS` 覆盖写
   - 如果写入过程中失败，正式文件可能变成截断文件或损坏文件

3. **恢复链缺少“未正式保存内容”的兜底层**
   - 当前只能恢复上次成功落盘的正式文件
   - 不能恢复“最后一次自动保存之后、崩溃之前”的编辑内容

### 当前保存安全路线

为保证数据安全，后续保存能力按下面顺序推进，不再把它们视为可长期拖后的增强项：

1. **Plan 6：建立保存安全内存模型**
   - 先明确工作文本、已落盘基线、dirty 三者关系

2. **Plan 7：自动保存**
   - 在统一保存模型之上按空闲、最长间隔和未落盘压力触发正式保存
   - 把“只在退出时保存”改成“平时就持续落盘”
   - 采用 **idle 保存 + 最长时间兜底 + 最大未落盘缓冲兜底**

3. **Plan 8：原子保存**
   - 把“直接覆盖正式文件”改成“临时文件 + 替换正式文件”
   - 只有完整成功后才推进已落盘基线
   - 避免保存失败时破坏既有正式文件

4. **Plan 9：崩溃恢复**
   - 为尚未进入正式基线的最近修改提供恢复层
   - 让应用在异常退出后尽可能恢复用户最近编辑内容

### 保存相关数据的层级约定

后续如果保存链扩展为“正式文件 + 恢复文件 / 日志 + 内存结构”，数据层级必须固定：

1. **`note.md` 是正式主文件**
   - 它是用户最终内容的正式落盘结果
   - 正常打开时，默认优先从它恢复

2. **恢复文件 / 恢复日志是辅助恢复层**
   - 它只服务“异常退出后的最近内容找回”
   - 它不能取代正式主文件成为长期主数据
   - 正常保存完成后，应被清理、折叠或重新开始

3. **AST 不是持久化真源**
   - AST 只是从文本推导出来的派生结构
   - 它可以缓存，但不能作为最终恢复真源
   - 真正的恢复真源必须仍然是文本主链

### 当前推荐的安全实现顺序

当前推荐先采用下面这条更稳妥的路线：

1. dirty
2. idle 自动保存
3. 定时兜底保存（最长间隔）
4. 最大未落盘缓冲兜底
5. 原子保存
6. 独立恢复文件

恢复日志 / journal 可以作为更后续增强，但**不作为当前第一版保存安全的必要前提**。

原因是：

1. 单便签、单窗口阶段，完整 journal 会明显提高实现复杂度
2. journal 需要 replay、截断、压缩、损坏处理等额外机制
3. 当前更需要先解决“丢数据”和“写坏正式文件”这两个核心风险

### 大段粘贴 / 大突发修改的处理原则

自动保存不能只依赖时间，还要考虑“未落盘修改量”。

当前推荐：

1. **idle 保存**：默认主策略
2. **最长时间兜底**：用户一直连续输入时触发
3. **最大未落盘缓冲兜底**：当自上次成功保存以来的新增 / 变更内容超过阈值时，提前触发保存

这里的“最大未落盘缓冲”不指 AST 大小，而指：

- 自上次成功保存以来
- 当前文本相对已落盘版本的未落盘变更量

它的目的主要是处理：

- 一次性大段粘贴
- 输入法一次性提交长文本
- 外部批量注入文本

处理原则是：

1. 超过阈值后，不必每次继续输入都立刻保存
2. 而是将保存请求提升为“尽快保存”
3. 如果此时用户仍持续输入，可使用更短的延迟窗口快速合并一次写盘
4. 仍不能退化成“每个字符都同步写盘”

### 保存安全完成线

保存安全主线至少做到下面 4 点，才算达到“笔记软件可接受”的最低线：

1. 普通编辑过程中，不必依赖用户主动关闭窗口才触发保存
2. 写盘失败时，旧正式文件尽量保持可读可用
3. 异常退出后，下次启动能检测并恢复最近未正式保存的内容
4. 上述能力仍严格走 `app -> storage` 主链，不打穿分层

## 当前不做

- 多窗口
- journal / checkpoint
- 跨平台实现
- Markdown 富文本渲染
- 复杂数据结构优化
