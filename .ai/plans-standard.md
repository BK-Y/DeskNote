# DeskNote 计划编写标准（Plans Standard）

## 1. 每个计划必须包含的章节

所有计划文件（`.ai/phases/*.md`）必须按以下结构编写：

```markdown
# 计划标题

## ① 核心问题
（为什么要做这个计划？当前缺失了什么行为或存在什么缺陷？）

## ② 目标
（验收时可观测的具体行为变化）

## ③ 方案设计
（方案对比与选择理由。如果只有一个方案，说明为什么没有备选）

## ④ 实现细节
（关键代码逻辑、接口签名、分层归属、约束条件）

## ⑤ 测试计划          ← 必填，不可省略
（见第 3 节「测试计划编写要求」）

## ⑥ 修改文件清单
（预计修改的文件列表，标注新增/修改/不应该修改的文件）

## ⑦ 推进步骤
（实施顺序，按时间排列的可执行步骤）

## ⑧ 风险与回退
（已知风险、缓解措施、回退方案）
```

## 2. 命名规范

### 2.1 文件命名

```
phase-{阶段编号}-shell-{子阶段}_{可选细分}.md
```

示例：
- `phase-10-shell-5a_repair-5-a-1_startup-register.md`
- `phase-10-shell-5b.md`

### 2.2 同一计划多个文件

若一个计划被拆分为多个子计划，子计划命名：

```
phase-{阶段编号}-shell-{子阶段}_{父计划名}-{序号}_{简短描述}.md
```

## 3. 测试计划编写要求    ← 核心标准

### 3.1 必须覆盖的维度

每个计划的测试计划（第⑤节）必须包含以下测试分组。缺一不可：

| 分组 | 目的 | 最低用例数 |
|------|------|-----------|
| **正常路径（Happy Path）** | 验证核心功能在理想条件下工作 | ≥ 1 |
| **边界条件（Edge Cases）** | 验证参数边界、空值、非法输入 | ≥ 1 |
| **错误处理（Error Handling）** | 验证失败路径行为（日志、状态保留、不崩溃） | ≥ 1 |
| **回归（Regression）** | 验证依赖的或相邻的已有功能未被破坏 | ≥ 2 |
| **集成（Integration）** | 验证跨子计划/跨层的组合场景 | ≥ 1 |

### 3.2 每个测试用例必须包含

```text
| 编号 | 用例名 | 前置条件 | 操作步骤 | 预期结果 | 覆盖需求 |
```

- **编号**：分组字母+序号（如 B-1、C-2）
- **前置条件**：state.ini 内容、窗口状态、菜单状态等
- **操作步骤**：具体的按键、菜单项、文件修改
- **预期结果**：可观测的行为 + 日志输出 + 文件内容变化
- **覆盖需求**：对应哪个计划或需求编号

### 3.3 日志验证要求

- 每个测试用例必须至少指定一条 **预期日志输出**（使用 OutputDebugStringW 产生的）
- 日志前缀必须在每个计划中一致，便于 DebugView 过滤
- 预期日志必须说明是 `成功`、`失败`、`跳过` 中的哪一种

### 3.4 测试数据准备

- 涉及 `state.ini` 的测试必须说明修改哪些字段、改成什么值
- 涉及旧格式兼容的测试必须说明"旧文件"的版本特征

### 3.5 验收标准

每个计划的验收标准必须是可观测、可重复的。以下句式禁止使用：
- ❌ "应该正常工作"
- ❌ "功能正确"
- ❌ "看起来没问题"

必须替换为：
- ✅ "DebugView 中出现 `[prefix] xxx` 日志"
- ✅ "state.ini 中 `key=value`"
- ✅ "窗口位置在屏幕 (`left`, `top`) 处"
- ✅ "编译通过"

## 4. 分层合规声明

每个计划的 **实现细节** 章节必须包含一段分层核实说明，例如：

```
分层核实：
- app 层仅调用 appbar.h 公开接口，不访问 s_appbar 内部字段
- platform 层不调用 app 层函数，无反向调用
- 本计划不修改：src/render/*、src/editor/*、src/core/*
```

## 5. 评审清单

提交 PR 前，自检以下项目：

- [ ] 第⑤节「测试计划」包含正常路径、边界、错误、回归、集成 5 个分组
- [ ] 每个测试用例有具体的预期日志输出
- [ ] 验收标准不包含模糊用语
- [ ] 分层合规声明已检查
- [ ] 修改文件清单与实际 diff 一致
- [ ] 已确认编译通过

## 6. 问题修复流程（快速模式）

### 6.1 适用场景

遇到 bug 或异常行为时，不走完整的 repair 计划流程，改用 debug_record 模式：

```text
发现问题 → debug_record → 直接改代码 → 更新记录
```

### 6.2 debug_record 写法

文件命名：`{关联计划名}_debug_record.md`

只写必要的四块内容，不需要模板式的 8 个章节：

```markdown
# xxx_debug_record — 问题简述

## 问题描述
（现象、触发条件、频率）

## 复现步骤
（从启动开始的精确操作序列）

## 调试过程
（按时间顺序列出：怀疑点 → 排查动作 → 结论/效果）

## 当前状态
（根因推测、待试方向、关联计划）
```

### 6.3 与 repair 计划的区别

| 维度 | repair 计划 | debug_record |
|------|-----------|-------------|
| 目的 | 规划新功能或结构性修复 | 记录调查过程和当前理解 |
| 前置条件 | 需要方案设计、测试计划 | 不需要，写到哪算哪 |
| 用时 | 写计划 10-30 分钟 | 边查边写，3-5 分钟 |
| 代码改了哪里 | 在计划中预先列出 | 改完后在记录中注明 |
| 什么时候创建 | 动手之前 | 发现问题时立即创建 |

### 6.4 注意事项

- debug_record 不需要完整模板，不填固定章节
- 修复成功后不需要把 debug_record 转为 repair 计划
- 如果问题长期未修复，可以在记录顶部加 `**状态：待修复**` 标记
- 多次尝试未果时，在「调试过程」中追加新行即可，不删历史
