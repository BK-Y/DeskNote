# Shell-3 — 便签壳层：托盘后台与窗口生命周期（后续）

## ① 核心问题

关闭窗口就退出程序了，不能后台常驻。重启后窗口状态全部丢失。

具体堵着的地方：
1. 窗口关闭 = 进程终止，没有办法"隐藏到后台继续运行"
2. 没有托盘图标，用户从系统托盘找不到入口恢复窗口
3. 窗口的 presence 状态（前台/隐藏）没有任何管理
4. 关闭行为没有区分"真的退出"和"只是隐藏到后台"
5. 重启应用后，上次的窗口状态（位置、显示模式）全部丢失

## ② 分析现状

### 承接关系

前置阶段（已完成）：

- `Shell-2b_done` — 窗口布局基线 + 四边四角缩放

本阶段消费 Shell-2b 的产物：

- frameless 窗口、标题栏、菜单按钮、拖拽、缩放
- 240×388 默认尺寸、屏幕右上角初始位置

### 阶段摘要

把系统从"只能以前台窗口存在"推进到"可隐藏到托盘后台运行、关闭/恢复/退出策略明确、状态可持久化"的状态。

## ③ 方案设计

### 候选方案

**方案 A：整体实现，不拆子阶段**
- 在一个阶段内一次性完成托盘创建、消息处理、presence 管理、关闭策略、持久化
- 优点：开发快，不需要多次上下文切换
- 缺点：单次改动量过大，难以定位问题；违反严格解耦原则——托盘消息、presence 状态、持久化是三个不同的职责
- 架构合规：不符合——三个不同职责应分开，否则 app 层会变成大杂烩

**方案 B：按职责拆为 5 个子阶段，逐层推进** [selected]
- 先做托盘创建（3a_1），再接入消息（3a_2），再定义状态（3b），再区分关闭策略（3c_1），最后持久化（3c_2）
- 理由：三个职责（平台API调用、状态管理、持久化）天然分属 platform/app/storage 三层，拆开执行符合分层约束；3a_1 创建图标后可独立验证，不阻塞后续
- 架构合规：符合——每个子阶段只改自己职责范围内的文件，分层清晰

**方案 C：使用第三方库（如 Qt SystemTrayIcon）**
- 引入 Qt 或 wxWidgets 的系统托盘抽象
- 优点：跨平台、API 统一
- 缺点：当前项目不依赖 Qt，引入重型依赖不合理
- 架构合规：不符合——严重违反当前技术栈约定

### 整体目标

1. 应用可隐藏到托盘后台运行，不占用任务栏位置
2. 从托盘可恢复窗口到前台
3. 关闭行为默认是隐藏到托盘而非退出
4. 标题栏汉堡菜单"关闭"改为隐藏到托盘，"关于"保持当前行为不变
5. 重启程序后恢复上次的窗口状态和位置

### 阶段产出

1. 系统托盘中出现 DeskNote 图标
2. 左键点击托盘图标 → 窗口恢复前台，右键 → 弹出菜单
3. 窗口可隐藏到托盘（任务栏图标消失），从托盘可恢复
4. **退出入口唯一化**：点窗口关闭按钮、Alt+F4、标题栏菜单"关闭"都隐藏到托盘；**唯一退出入口 = 托盘右键菜单"退出"**
5. 重启程序后窗口位置和显示模式恢复

## ⑤ 设定边界

### 本阶段范围

- 托盘图标的创建、事件处理、弹出菜单
- presence 状态（visible_front / hidden_to_tray / exiting）定义与转换
- 关闭按钮行为区分
- 状态持久化到 state.ini

### 不包括

- 不支持最小化到任务栏（便签不需要任务栏图标）
- 不做 UWP 或停靠区域交互
- 不做 floating topmost（那是 Shell-4）
- 不做 AppBar / 贴边占位（那是 Shell-5）

### 约束条件

1. hidden_to_tray 时任务栏图标必须消失
2. visible_front 时任务栏正常显示
3. 关闭策略的业务判断留在 app，platform 只执行窗口显隐
4. 不支持最小化到任务栏

### 分层归属

| 层 | 职责概要 |
|------|---------|
| platform/win32/window | 托盘图标创建/销毁、WM_TRAYICON 消息处理、ShowWindow 配合显隐、WM_CLOSE 拦截 |
| app/ | 声明 ShellPresenceState、托盘生命周期接口、presence 转换逻辑、关闭策略分发 |
| storage/ | state.ini 中读写 presence_state、resident_mode |
